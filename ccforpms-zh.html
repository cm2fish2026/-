📖 概览

📖 概览

Claude Code 是一个直接在你电脑上操作文件和文件夹的 AI 助手。与传统聊天机器人需要手动复制粘贴上下文不同，Claude Code 可以通过自然对话直接读取文档、分析数据、创建材料。

🎯 你将构建什么

本课程教你用 Claude Code 搭建一套 PM 生产力系统，包含以下核心能力：

  1. 文件操作——用 @ 符号读取、写入和引用文件
  2. 视觉分析——粘贴截图，分析 Mockup、图表、数据
  3. 并行 Agent——同时运行多个 AI 实例处理批量任务
  4. 自定义子 Agent——构建有个性的永久专属团队成员
  5. 项目记忆——通过 CLAUDE.md 维护关于产品和团队的持久上下文

0.1 安装配置

⏱ 15 分钟前提：Claude Pro 或 Max 订阅
核心要点：Claude Code 需要 Node.js 18+ 和付费 Claude 订阅。前提条件满足后，安装用时不到 5 分钟。

如何打开终端

Mac：按 Cmd + Space 打开 Spotlight，输入「Terminal」，按回车。

Windows：按 Windows 键，输入「PowerShell」，按回车。

检查 Node.js 版本

在终端中输入以下命令并按回车：

node --version

如果看到 v18.0.0 或更高版本，可直接进入安装步骤。否则请先安装 Node.js。

安装 Node.js（如需要）

推荐方式：从 nodejs.org 下载 LTS 版本。

平台命令
macOSbrew install node
Windowschoco install nodejs

重要：安装 Node.js 后，完全关闭终端窗口并重新打开，以刷新系统 PATH。

安装 Claude Code

Mac：

curl -fsSL https://claude.ai/install.sh | bash

Windows（PowerShell）：

irm https://claude.ai/install.ps1 | iex

验证安装

claude --version

如果看到版本号，安装成功。如果提示「command not found」，完全关闭终端并重新打开后再试。

认证

在终端输入 claude 并按回车，然后：

  1. 选择「Claude account with subscription」（不是 Anthropic Console）
  2. 完成浏览器认证
  3. 选择终端文字样式

认证完成后，你会看到 Claude Code 欢迎界面。

健康检查

在 Claude Code 内部输入：

/doctor

该命令会验证安装完整性、有效订阅、系统配置，健康状态所有条目会显示绿色勾选。

常见问题排查

提示「command not found: claude」

完全关闭终端并重新打开。如仍然失败，检查 PATH 配置：

Mac：在 ~/.bashrc 或 ~/.zshrc 中添加：

export PATH=$HOME/.local/bin:$PATH

Windows：重启电脑（安装程序会自动更新 PATH）。

提示需要订阅

在 claude.ai 验证有效的 Claude Pro/Max 订阅，然后退出重新登录。

Node 版本过低

检查版本：

node --version

如低于 v18.0.0，从 nodejs.org 下载最新 LTS 版本。

⏱ 15-20 分钟

核心要点：可视化工作区让你能实时看到文件在 Claude 工作时出现和更新。

为什么可视化很重要

在终端中使用 Claude Code 感觉像蒙眼工作——你让 Claude 创建文档，却看不到正在发生什么。解决方案是在终端旁边打开一个可视化编辑器。这样你就可以实时看到文件的创建与更新，随时了解正在发生什么，把 Claude Code 从神秘的黑盒变成透明可视的工具。

选择你的工具

Nimbalyst（推荐）

Nimbalyst 是专为 Claude Code 用户打造的免费工具，具备所见即所得 Markdown 编辑、文件差异对比、隐藏文件夹可见性和会话管理等功能。

Obsidian

Obsidian 是免费的流行 Markdown 编辑器，界面简洁、无干扰。局限：无法显示 .claude/ 等隐藏文件夹。

VS Code / Cursor

功能完整的代码编辑器，可显示隐藏文件夹，适合已在使用这类工具的用户。

快速设置

Nimbalyst：下载并安装，打开课程文件夹，将 Claude Code 终端和 Nimbalyst 并排放置。

Obsidian：访问 obsidian.md/download 下载，选择「Open folder as vault」，导航到课程文件夹，并排放置。

隐藏的 .claude/ 文件夹

.claude/ 文件夹包含 Agent、命令和设置。不同工具的访问方式：

工具能否查看 .claude/访问方式
Nimbalyst能文件浏览器中直接可见
Obsidian否使用 Finder / 文件资源管理器
VS Code能启用显示隐藏文件

Mac：在 Finder 中按 Cmd + Shift + . 显示隐藏文件夹。

Windows：在文件资源管理器的「查看」选项卡中勾选「隐藏项目」。

最佳实践

  • 始终使用分屏工作流
  • 使用快速打开快捷键导航（Obsidian：Cmd/Ctrl + O）
  • 将编辑器聚焦在项目文件夹
  • 不要仅使用全屏终端——你会错过一半的价值

🤯 什么是 Agent？

Agent 是独立并行运行的 Claude 实例。把它想象成克隆自己来同时处理多个任务：

串行工作（无 Agent）：

任务1 (5分钟) → 任务2 (5分钟) → ... → 任务10 (5分钟)
总耗时：50 分钟

并行工作（有 Agent）：

任务1 ┐
任务2 │ 同时进行
任务3 ├─→ 全部并行
... │
任务10 ┘
总耗时：5 分钟

🎯 何时使用 Agent

📊 影响估算框架

影响 = 受影响用户数 × 当前行为率 × 预期提升 × 每次行为价值
  • 受影响用户数：有多少用户会看到这个功能？考虑逐步推出（并非总是 100%）
  • 当前行为率：当前有多少比例完成期望行为？从分析工具获取
  • 预期提升：功能能将比率提升多少？来源：历史数据、竞品基准、用研
  • 每次行为价值：每个增量行为值多少？激活用户 LTV、留存 LTV 等

三场景方法

场景采纳率提升用途
悲观（20 百分位）30%45% → 50%最低预期影响
现实（50 百分位）70%45% → 58%最可能情况
乐观（80 百分位）90%45% → 62% + 留存提升最佳情况

🔬 实验分析框架

  1. 顶线指标——计算对照组和实验组的整体激活率
  2. 统计显著性——p < 0.05 表示统计显著（随机概率 < 5%）
  3. 细分分析——按目标客户细分实验结果，顶线平均值可能掩盖细分市场的胜出
  4. 质量指标——已激活用户中，检查第 1 周留存率
  5. 领先指标——功能采用率、邀请率、参与深度

提示词黄金法则

法则 1：迭代，不要重来

如果图片 80% 正确，就请求具体的修改，而不是从头开始。每次改进都给 Gemini 更多关于你想要什么的信息；重新开始则丢弃所有这些上下文。

法则 2：使用自然语言和完整句子

用自然语言和 Gemini 对话，而不是关键词堆砌。

关键词堆砌（已过时）：

professional headshot, business attire, confident, studio lighting, bokeh, 8k

自然语言（更好）：

法则 3：具体描述细节

定义主体、场景、光线、情绪、纹理和材质。考虑颜色调色板、构图和焦点。Gemini 能处理比你想象中更多的细节。

法则 4：提供上下文（「为什么」和「为谁」）

上下文塑造创意选择。「用于儿童读物」的肖像与「用于奢侈品牌广告」的同一主体看起来完全不同。

什么是 Vibe Coding？

Vibe Coding 是通过描述你想要什么来构建软件的方式，而不是自己写代码。这个词由 Andrej Karpathy 推广——「完全沉浸在感觉中」。你描述结果，AI 负责实现。技能不在于编码，而在于清楚地说明你想要什么、给出好的反馈，以及判断什么时候结果是对的。

PM / 工程师动态

什么是需求？

需求是在构建之前对你要构建什么的描述——就像你与构建者之间的「合同」。细节越充分，后续意外就越少。无论你与 AI 工具、开发者、设计师还是代理机构合作，这个原则都适用。模糊指令 = 出乎意料的结果。具体需求 = 你真正想要的。

软件中的「逻辑」

逻辑是行为规则，是「如果这样，那么那样」的层次。每个应用都有逻辑：如果用户点击按钮 → 显示下一个屏幕；如果答案 = A → 给个性 X 加分；如果得分最高 → 显示该结果。你用白话文定义逻辑，Claude 用代码实现它。

4.3 构建与迭代

⏱ 30-40 分钟

开发服务器是什么？

开发服务器在本地运行你的代码，让你可以像网站一样预览它。它只在你的电脑上工作（非公开），需要在你查看网站时保持运行，用 Ctrl+C 停止。

localhost 是什么？

localhost 表示你自己的电脑作为服务器运行。localhost:3000 = 你电脑上的 3000 端口。这不是真实的 URL，其他人无法访问。你在 4.5 部署后才会获得真实 URL。

为什么迭代很重要

第一个版本永远不会完美——这是正常的，也是预期的。专业开发永远是：构建 → 审查 → 优化 → 重复。使用 AI 的不同之处：迭代只需几秒而非几天。所以不要将就——持续优化直到你满意为止。

截图技巧

在所有平台（包括 Mac！）用 Ctrl+V 直接粘贴图片到 Claude Code。这对迭代非常有力：截取有问题的地方 → 圈出问题或指向它 → 粘贴并描述你想要的不同效果。视觉反馈比用文字描述问题快得多。

常见问题排查

  • 「npm: command not found」——未安装 Node.js，从 nodejs.org 下载安装后重启终端
  • 「Port 3000 already in use」——关闭其他可能运行服务器的终端窗口
  • 浏览器未打开——手动访问 http://localhost:3000
  • 更改未显示——刷新浏览器（Cmd+Shift+R 或 Ctrl+Shift+R 清除缓存）

4.4 GitHub

⏱ 15 分钟
在 Claude Code 中运行 /start-4-4。
本课内容：将项目推送到私有 GitHub 仓库。

版本控制是什么？

版本控制是随时间追踪文件每次更改的系统，类似「Word 中的修订追踪」，但作用于整个项目。你可以查看所有更改的历史记录，回到任何之前的版本，撤销错误——永久解决「最终版_v3_真的是最终版.docx」问题。

Git vs GitHub

Git = 在你电脑上运行的版本控制系统，本地追踪更改。

GitHub = 在云端存储 Git 仓库的网站。

类比：Git 是 Word（工具），GitHub 是 Google Drive（存储和分享的地方）。

关键术语

术语含义
仓库（repo）被 Git 追踪的项目文件夹
提交（commit）某个时间点的保存快照
推送（push）从你的电脑上传提交到 GitHub
拉取（pull）从 GitHub 下载更改到你的电脑

为什么部署需要这个

你的代码目前只存在于你的电脑上。更重要的是：部署服务需要从某处拉取你的代码，而 GitHub 就是那个地方。下一课部署时，Vercel 会连接到你的 GitHub 仓库并自动拉取代码。

4.5 上线部署

⏱ 15 分钟

部署是什么？

部署是把你电脑上的代码放到互联网上的过程：你的代码 → 任何人都可以访问的服务器 → 真实 URL。开发者说「部署」、「上线」，意思都是这个。这是你的作品变成真实存在的时刻。

什么是 Vercel？

Vercel 是一个网站托管服务，个人项目免费，由 Next.js 的创建者打造，与 GitHub 集成，代码推送时自动部署。给 Vercel 你的 GitHub 仓库 → 它构建代码 → 你获得真实 URL，如 yourproject.vercel.app。

自动部署的魔法

  1. 你将代码推送到 GitHub
  2. Vercel 检测到更改
  3. Vercel 自动重建并重新部署
  4. 你的线上网站约 1 分钟内更新

初始设置后无需手动操作：改代码 → 推送 → 上线。

2. 项目与组织

Sarah 领导一家快速成长的 SaaS 公司的 IT 运营，管理 800 名员工使用的 15 个工具，年预算 200 万美元，直接汇报给 CTO。从 IT 支持专员成长到现在，拥有 CISSP 认证。

目标与动机

  • 减少工具泛滥——从 25+ 个工具整合到 10-12 个
  • 改善安全态势——通过 SOC 2 审计
  • 控制成本——需要对每个工具证明 ROI
  • 赋能团队——在不妥协安全的前提下提供好工具

痛点

  • 工具泛滥：工程用 Linear，市场用 Monday.com，销售用 Salesforce+Asana……
  • 安全噩梦：影子 IT、无法看到谁有什么权限、离职员工仍可访问工具
  • 入职/离职混乱：新员工需手动创建 25 个账号，2 天才能完成

关注的功能

必备：SSO（SAML）、审计日志、高级权限、批量用户管理、管理员仪表板

不关心：花哨的 UI 动效、社交功能、移动应用

标志性引言

「安全不容商量。如果你没有 SSO 和审计日志，我们甚至不会考虑你。」

项目详情
姓名Mike Rodriguez，29 岁
职位高级软件工程师
公司GrowthLabs（150 人，B 轮）
工作方式每天编码 6-8 小时，每周 2-3 次会议

目标与动机

  • 深度专注——长时间不被打断的专注工作
  • 清晰的优先级——确切知道要做什么，无需猜测
  • 最小化上下文切换——更少的工具，减少跳转
  • 交付高质量代码——对工艺有自豪感

痛点

  • 优先级混乱：多个「真相来源」（Slack、邮件、站会、任务工具）
  • 上下文不足：任务标题「修复登录 bug」——哪个 bug？在哪里？为什么？
  • 会议泛滥：站会本可以是异步的，状态更新打断了专注
  • 工具缓慢臃肿：页面加载超过 3 秒，点击太多才能完成简单操作

关注的功能

必备：快捷键、快速加载、GitHub 集成、完整 Markdown（含代码块）、任务上下文清晰

加分项：暗黑模式、离线模式

不关心：视觉项目看板、图表报表

标志性引言

「只要告诉我要构建什么，给我上下文，然后让我专注写代码。不要让我去找信息或坐在会议里。」

关注的功能

必备：团队仪表板、工作负荷视图、阻塞任务可见性、冲刺报告、评论摘要

不关心：细粒度任务详情（信任团队处理）

标志性引言

「我需要知道我的团队是否在正轨上，而不必每天单独询问他人。这不可持续。」

主要竞品深度分析

Asana

优势：市场领导者，品牌认知度高，200+ 集成，企业级功能完备，支持多种视图。

弱势：UI 复杂，学习曲线陡峭；Business 方案 $30/用户/月；页面加载慢；功能过度工程化；工程师不喜欢。

我们的差异化：更简单（更快上手）、更便宜（同等功能价格低 50%）、更快（性能更好）、更现代（更好的设计）。

什么时候我们赢：厌倦复杂的初创公司、想要更好 UX 的团队、注重预算的公司、想要开发者友好工具的工程团队。

Linear

优势：同类产品中最好的设计，开发者文化的最爱，闪电般的性能，最佳 GitHub 集成，有主见但不让人迷失。

弱势：专注工程，不适合跨职能团队；缺少 PM 功能（无甘特图、报表有限）；集成较少。

我们的差异化：跨职能（适合整个产品团队）、更多功能（甘特图、资源规划）、异步优先（更好的分布式团队支持）。

什么时候我们赢：跨职能团队、PM 主导工具选择、需要 PM 专属功能、分布式团队。

我们的市场定位

我们的定位：面向远程初创公司的现代项目管理工具

我们的优势：均衡（不太简单也不太复杂）、异步优先 DNA（原生构建，非后来添加）、开发者和 PM 都喜欢、快速且精致。

我们的劣势：只有 3 年历史，集成数量（50 vs Asana 的 200+），品牌知名度较低。

赢单/败单分析（上季度）

我们赢了

  • vs Asana（12 单）：太贵、太复杂、更好的开发者体验
  • vs Linear（8 单）：需要跨职能功能、PM 是采购方、需要报表
  • vs ClickUp（10 单）：性能问题、界面杂乱、追求质量而非功能数量

我们输了

  • 输给 Asana（5 单）：企业要求成熟供应商、已有 Asana 大量投入、特定高级功能缺失
  • 输给 Linear（7 单）：工程团队主导选型（他们爱 Linear）、极度追求 UI 美观
  • 输给 Notion（4 单）：偏好一体化（文档 + 任务）、已在使用 Notion 写文档

需要关注的竞争趋势

  1. AI 集成——所有竞品都在添加 AI 功能（Notion AI、Asana Intelligence），我们需要 AI 策略
  2. 移动优先——Linear 正在打造精美移动应用，我们也需要（Q1 2025）
  3. 一体化趋势——Notion、ClickUp 在扩大范围，我们该添加文档/Wiki 吗？
  4. 向上拓展企业市场——Linear、Notion 都在往上走，我们也应该（SSO、权限）

关键结论

从竞品学什么：Linear 告诉我们 UI 质量和速度至关重要；Asana 告诉我们企业功能是向上市场的必要条件；Notion 告诉我们灵活性吸引用户。

什么是我们的可持续优势：异步优先 DNA（与生俱来，非外加）、均衡定位（简洁与强大的正确结合）、快速（性能很重要）、初创公司文化契合。
